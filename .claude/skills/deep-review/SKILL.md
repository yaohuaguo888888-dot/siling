---

## 深度审查执行步骤

### 第一遍：变更理解
- 阅读PR描述和关联Issue，理解变更意图
- 通读所有diff，建立变更全貌

### 第二遍：逻辑正确性
- 逐文件检查：变量类型、空值处理、边界条件、异常路径
- 跨文件检查：函数签名变更 → 搜索所有调用方是否同步
- 状态流转：状态变量修改 → 追踪所有读取该状态的代码路径

### 第三遍：系统影响分析
- 识别被修改模块的下游消费者
- 检查隐含契约是否被破坏（返回值格式、副作用、时序假设）
- 评估并发场景下的安全性

### 第四遍：架构合规
- 检查依赖方向（不允许下层依赖上层）
- 验证分层边界（UI层不直接访问数据层等）
- 确认设计模式使用一致性

### 输出格式
```
## 审查结果

### Critical Issues
- [文件:行号] 描述 | 影响: ... | 修复: ...

### Major Issues
- [文件:行号] 描述 | 影响: ... | 修复: ...

### Minor Issues / Suggestions
- [文件:行号] 描述 | 建议: ...

### 总体评估
- 可合并性: ✅/⚠️/❌
- 风险等级: 低/中/高
```
